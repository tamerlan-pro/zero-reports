# Processes

Все бизнес-процессы системы. Описаны от начала до конца, без разбивки по модулям.

---

## 1. Онбординг артиста

**Кто:** Менеджер лейбла, Система

От создания артиста до полной готовности к получению выплат.

1. Менеджер создаёт артиста (nickname, legal_name)
2. Система создаёт привязку артиста к лейблу
3. Система автоматически создаёт пустую запись ArtistRegistration
4. Система запускает чеклист полноты данных
5. Система генерирует Task по каждому незаполненному пункту:
   - Сплиты проставлены
   - Договор подписан
   - IMRO подключено
   - Tipalti / EasyStaff подключено
   - Доступ к личному кабинету выдан
   - Стартовое сообщение отправлено
   - Формы подписаны
6. Менеджер работает по списку: заполняет данные, связывается с артистами
7. При заполнении каждого пункта система автоматически закрывает соответствующий Task
8. Менеджер продолжает, пока все статусы не станут зелёными
9. Все Task закрыты → артист полностью готов к выплатам

**Периодичность:** при подписании нового артиста.

## 2. Управление сплитами

**Кто:** Менеджер лейбла

### 2a. Создание или изменение сплита

1. Менеджер находит релиз (по UPC) или трек (по ISRC)
2. Менеджер создаёт или редактирует сплит — указывает участников и их доли
3. Система проверяет сумму долей = 100%
4. Если сумма ≠ 100% → отклонить
5. Система сохраняет новую версию сплита (effective_from = today)
6. Предыдущая версия получает effective_to = today (история сохраняется)

**Периодичность:** по мере необходимости.

### 2b. Ретроактивное изменение сплита (добавление MC задним числом)

Это главная бизнес-боль. Типичный кейс: трек уже зарабатывает, и задним числом добавляется MC (вокалист/рэпер), который ранее не был учтён.

1. Менеджер добавляет нового участника (MC) к существующему сплиту
2. Менеджер указывает, у кого из текущих участников забирается доля
3. Система создаёт новую версию сплита
4. Система рассчитывает ретро-пей: для каждой транзакции за весь период вычисляет, сколько бы MC заработал
5. Система показывает превью: сколько кому начислится/спишется
6. Менеджер подтверждает пересчёт
7. Система создаёт Adjustment-записи: (+) для MC, (−) для тех, у кого доля уменьшилась
8. Проверка: SUM(adjustments) = 0 (баланс сходится)
9. Если MC не зарегистрирован в платёжной системе → создать Task менеджеру на подключение MC к Tipalti/EasyStaff

**Периодичность:** несколько раз в месяц.

## 3. Импорт отчётов платформ

**Кто:** Финансовый менеджер, Система

Каждая платформа имеет свой формат и свою стратегию привязки треков. Ключевой принцип: каждый ручной маппинг создаёт TrackAlias, поэтому с каждым импортом ручной работы становится меньше.

### 3a. Дистрибьюторы (SoundOn, Believe) — привязка по ISRC

1. Финансовый менеджер загружает файл отчёта
2. Система определяет платформу по формату
3. Система парсит файл — каждая строка становится Transaction
4. Система привязывает Transaction к Track по ISRC (прямое совпадение)
5. Если ISRC не найден — Transaction помечается как непривязанная
6. Система определяет Label через Track → Release → Label
7. Система применяет действующую версию Split
8. Система показывает результат: привязанные / непривязанные / ошибки
9. Финансовый менеджер проверяет и подтверждает
10. Система обновляет балансы

**Периодичность:** ежемесячно.

### 3b. ASCAP (Domestic + International) — привязка по Work ID или названию

1. Финансовый менеджер загружает файл отчёта ASCAP
2. Система определяет тип: Domestic или International по формату
3. Система парсит файл — каждая строка становится Transaction
4. Система связывает юридическое имя автора с артистом в каталоге
5. Система привязывает Transaction к Track: сначала по Work ID → TrackAlias, затем по нормализованному названию
6. Если трек не найден — Transaction помечается как непривязанная
7. Финансовый менеджер вручную маппит → система создаёт TrackAlias
8. Система применяет Split и обновляет балансы

**Периодичность:** ежеквартально.

### 3c. B17 (Sony Music Publishing) — привязка по названию + автору

1. Финансовый менеджер загружает файл отчёта B17
2. Система парсит файл
3. Система связывает Songwriter с артистом по юридическому имени
4. Система привязывает Song к Track по нормализованному названию → TrackAlias
5. Если не найден — вручную маппит, система создаёт TrackAlias
6. Система применяет Split и обновляет балансы

**Периодичность:** ежеквартально.

### Общий Revenue Flow (после импорта)

```
Файл отчёта платформы
  ↓
Парсинг → Transaction
  ↓
Привязка Transaction к Track (по ISRC / Work ID / названию)
  ↓
Если не найден → unmatched → ручной маппинг → создание TrackAlias
  ↓
Разбивка по Label через Track → Release → Label
  ↓
Применение Split версии по period_date → распределение дохода по участникам
  ↓
Обновление Balance
  ↓
Данные доступны в отчётах
```

## 4. Рекламные расходы

**Кто:** Рекламный менеджер / Система (из промо-пайплайна)

### 4a. Ручной ввод расхода

1. Менеджер выбирает релиз или трек
2. Менеджер вносит сумму расхода, период и описание
3. Менеджер устанавливает флаг recoup (true = вычитается из дохода артиста, false = no recap)
4. Система сохраняет AdExpense
5. Баланс артиста пересчитывается

### 4b. Автоматический из промо-пайплайна

1. Система агрегирует PromoVideo за период
2. Фильтрует дубликаты, удалённые, недоступные
3. Проверяет PromoRecognition: confirmed/failed → принимаются, mismatch → на проверку
4. Группирует: PromoTrack → Track → Release → Artist
5. Рассчитывает суммарный расход по треку/релизу
6. Формирует предварительный список AdExpense
7. Head of Marketing проверяет и подтверждает
8. Система создаёт AdExpense
9. Система пересчитывает балансы

**Периодичность:** ручной — по мере необходимости, автоматический — ежемесячно (перед выплатами).

## 5. Ежемесячный цикл выплат (Payout)

**Кто:** Финансовый менеджер, Владелец/Директор, Система

1. Система рассчитывает "Ready to Pay" по каждому артисту:
   `balance = income − recouped_expenses − advances − paid_payouts`
2. Финансовый менеджер формирует два списка: Tipalti (иностранные) и EasyStaff (RU)
3. Списки сортируются по убыванию суммы
4. Незарегистрированные артисты с крупными балансами → отдельный список + Task на подключение
5. Финансовый менеджер отправляет списки на согласование владельцу/директору
6. Владелец/Директор проверяют суммы, уточняют отдельных артистов, вносят корректировки
7. Владелец пополняет баланс Tipalti/EasyStaff
8. **Tipalti:** финменеджер загружает batch-файл (CSV с IDAP-кодами и суммами)
9. **EasyStaff:** финменеджер создаёт таски на каждого артиста
10. Для EasyStaff: артисты принимают таски, кто не принял — пушат через менеджеров
11. Через 3–5 дней: пост-контроль статусов (Submitted → Paid / Error)
12. Система фиксирует все выплаты, обновляет балансы

Фильтрация: исключаются артисты с bought off rights, нулевым балансом и незарегистрированные в платёжной системе.

**Периодичность:** ежемесячно, ~1–17 числа.

## 6. Промо-пайплайн

### 6a. Создание кампании и распределение бюджетов

**Кто:** Head of Marketing, Тимлид

1. Head of Marketing создаёт PromoCampaign (название, период, лейбл)
2. Head of Marketing добавляет треки — по каждому создаётся PromoTrack:
   - выбирает Track из каталога (по ISRC или названию)
   - указывает Sound ID для неоригинальных звуков
   - устанавливает priority, geo, categories, comment, total_budget
3. Head of Marketing создаёт PromoTrackAllocation: команда + менеджеры + бюджет
4. Один трек может иметь несколько аллокаций (разные команды/менеджеры)
5. Head of Marketing утверждает кампанию → статус Active
6. Тимлид видит свои треки и бюджеты
7. Система отслеживает расход: team_spent vs team_budget
8. При исчерпании бюджета → Head of Marketing решает

**Периодичность:** 1–2 раза в месяц.

### 6b. Сделки с блогерами

**Кто:** Промо-менеджер, Тимлид

1. Промо-менеджер находит или создаёт Blogger
2. Промо-менеджер создаёт BloggerDeal через Telegram бот (количество, ставка, платформа)
3. Тимлид подтверждает сделку (если сумма выше порога)
4. Промо-менеджер инициирует оплату → BloggerPayment
5. Оплата проходит → статус Paid
6. BloggerDeal → Confirmed → ожидание роликов
7. По мере выхода роликов система обновляет прогресс
8. Все ролики выпущены → Fulfilled; часть не выпущена → эскалация тимлиду

**Периодичность:** ежедневно.

### 6c. Подача промо-роликов

**Кто:** Промо-менеджер

1. Промо-менеджер кидает ссылку на ролик в Telegram бот
2. Система парсит: video_id, profile_id, платформу
3. Система проверяет дубликаты
4. Менеджер ВЫБИРАЕТ PromoTrack из активной кампании (ключевое отличие — устраняет "чёрный ящик")
5. Менеджер указывает rate и (optional) BloggerDeal
6. Система создаёт PromoVideo со связями
7. Система запускает верификацию (Audd/Shazam) → PromoRecognition
8. Если mismatch → уведомление тимлиду
9. Если BloggerDeal указан → обновляет прогресс сделки

**Периодичность:** непрерывно (десятки роликов в день).

### 6d. Мониторинг долга

**Кто:** Система, Тимлид, Head of Marketing

1. Система сопоставляет по менеджеру: ∑ оплачено (BloggerDeal) vs ∑ вышло (PromoVideo)
2. Рассчитывает долг в $ и штуках
3. Сопоставляет по блогеру: оплачено vs вышло
4. Обновляет дашборд долга (ранкинги менеджеров и блогеров)
5. Личная страница менеджера: его блогеры с долгами
6. При долге выше порога → тимлиду уведомление
7. Тимлид разбирает: блогер не выпустил → пушит, менеджер не подал → пушит
8. Head of Marketing видит суммарный долг

**Периодичность:** непрерывно (обновляется при каждом PromoVideo и BloggerPayment).

### 6e. Расчёт перформанса и бонусов

**Кто:** Система, Head of Marketing

1. Система агрегирует PromoVideo по менеджеру за период
2. Рассчитывает KPI: Releases, Бюджет, Reach, Views/$
3. Рассчитывает Score и Bonus
4. Агрегация по PromoTeam
5. Рассчитывает TL Bonus
6. Формирует ранкинг менеджеров
7. Обновляет Overview, Per-Manager, Weekly/Monthly дашборды
8. Head of Marketing принимает решения по командам

**Периодичность:** понедельно (автоматически), бонусный расчёт — ежемесячно.

## 7. Генерация отчётов

**Кто:** Менеджер лейбла / Артист

### 7a. Менеджер генерирует отчёт

1. Менеджер выбирает артиста или вводит UPC
2. Система агрегирует данные:
   - Доходы по трекам и платформам (из Transaction)
   - Текущие и исторические доли (из Split)
   - Баланс, расходы, авансы, выплаты (из Finance)
3. Система формирует отчёт с детализацией
4. Менеджер просматривает и при необходимости выгружает в файл

### 7b. Артист смотрит сам

1. Артист открывает свой кабинет
2. Система показывает: доходы по трекам и платформам, расходы, выплаты, баланс
3. Артист может запросить выгрузку транзакций
4. Артист видит источники дохода и периоды без необходимости писать менеджеру

**Периодичность:** по запросу.

---

## 8. Финансовый дашборд

**Кто:** Владелец компании

```
По каждому Label:
  ├── Доходы по платформам (Believe | SoundOn | ASCAP | B17)
  ├── Рекламные расходы (AdExpense)
  ├── Авансы (Advance)
  └── Выплаты (Payout)
  ↓
Группировка по источникам
  ↓
Сводка → дашборд владельца
```

**Периодичность:** доступен в любой момент, данные обновляются при каждом импорте/выплате.
